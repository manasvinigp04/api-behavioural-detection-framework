OVERVIEW
═══════════════════════════════════════════════════════════════════

This is a RESEARCH-GRADE system for detecting multi-dimensional API drift
using hybrid approaches (symbolic + statistical + machine learning).

NOVEL CONTRIBUTIONS:
1. Multi-fidelity contract fusion (OpenAPI + PRD)
2. Hybrid drift detection (3-layer architecture)
3. Behavioral modeling from incomplete specs
4. Progressive drift tracking (time-series)
5. Explainable drift analysis (confidence scoring)

═══════════════════════════════════════════════════════════════════
DATA FLOW
═══════════════════════════════════════════════════════════════════

1. INPUT PROCESSING
   OpenAPI Spec → OpenAPIParser → UnifiedSpec
   PRD Document → PRDParser → InferredContract (with confidence)
   ─────────────────────────────────────────────────────────────→
   ContractBuilder.merge() → ICR (Unified Contract)

2. TEST GENERATION
   ICR → ValidGenerator   → Valid test cases
      → InvalidGenerator → Invalid test cases
      → BoundaryGenerator → Boundary test cases
      → SemanticGenerator (LLM) → Semantic adversarial cases
   ─────────────────────────────────────────────────────────────→
   RiskRanker → Prioritized TestSuite

3. EXECUTION
   TestSuite → ParallelExecutor (10 workers) → Live API
   ─────────────────────────────────────────────────────────────→
   ResultCollector → ExecutionResults

4. DRIFT DETECTION (Hybrid)
   ExecutionResults → ContractDriftDetector (Symbolic)
                   → ValidationDriftDetector (Symbolic)
                   → BehavioralDriftDetector (Statistical + ML)
                   → ProgressiveDriftDetector (Time-series)
   ─────────────────────────────────────────────────────────────→
   DriftScoreAggregator → DriftReport

5. EXPLAINABILITY
   DriftReport → RootCauseAnalyzer (Pattern + LLM)
   ─────────────────────────────────────────────────────────────→
   ExplanationGenerator → AnalysisResult

6. REPORTING
   AnalysisResult → MarkdownGenerator → report.md
                 → JSONGenerator → report.json
                 → CLIFormatter → terminal output

═══════════════════════════════════════════════════════════════════
KEY DESIGN DECISIONS
═══════════════════════════════════════════════════════════════════

1. MULTI-FIDELITY INPUT
   WHY: APIs often have incomplete/outdated specs. Combining structured
        (OpenAPI) and unstructured (PRD) sources improves coverage.
   HOW: Confidence-scored merging. OpenAPI=1.0, PRD=0.6-0.9.

2. HYBRID DETECTION
   WHY: Symbolic has high precision but low recall. ML has high recall
        but low precision. Hybrid balances both.
   HOW: Three layers - symbolic → statistical → ML. Each layer adds
        drift types the previous layers miss.

3. BEHAVIORAL MODELING
   WHY: When specs are incomplete, learn expected behavior from history.
   HOW: Embed responses, cluster similar behaviors, detect anomalies.

4. PROGRESSIVE DRIFT
   WHY: APIs drift gradually. Single-snapshot detection is insufficient.
   HOW: Time-series analysis with change point detection and forecasting.

5. EXPLAINABILITY
   WHY: Developers need actionable insights, not just "drift detected".
   HOW: Pattern matching + LLM-assisted causal inference + code examples.
